## Context

task-board は個人ローカル用のカンバン式タスク管理 Web アプリの新規プロジェクトである。本変更 `initial-kanban-mvp` は、このプロジェクトにおける最初の変更提案であり、以下を同時に定める必要がある。

- 実行形態 (どのようにユーザーが起動し、どのように動くか)
- 技術スタック (フレームワーク、言語、ライブラリ)
- データ永続化の方式と保存先
- 書き込み系 API の構成
- ドラッグ & ドロップの実装方針
- 板・カード・カラムの初期モデル

これらは以降のすべての機能追加に影響するため、判断理由 (Why) と却下案を残しておく必要がある。本ドキュメントは初回の意思決定を ADR (Architecture Decision Record) 形式で記録することを主目的とする。

前提となる制約:
- 利用者は自分ひとり (シングルユーザー・ローカルのみ)
- 認証・マルチユーザー・リモート同期は対象外
- ブラウザ制約 (File System Access API 等の縛り) を避けたい
- 「起動する」ステップ自体は許容できる

参照: `proposal.md` (Why / What / Non-goals)

## Goals / Non-Goals

**Goals:**
- 個人ローカル用途で、最小限の摩擦で日常的に使える MVP を成立させる。
- 技術判断の理由・却下した選択肢を ADR として残し、将来の再検討 (例: JSON → SQLite 移行、DnD ライブラリの差し替え) を容易にする。
- 3 列固定カンバン + タイトルのみカード + DnD 移動、という最小仕様を実装可能な粒度まで分解する。
- 単一プロセス (Next.js のみ) で UI とデータ永続化を完結させる。

**Non-Goals:**
- MVP の段階でスケーラビリティ・並行編集・障害耐性を確保すること。
- ブラウザ非依存 (Firefox / Safari 完全対応の検証) を保証すること。動作対象は「使う人の使うブラウザ」で、まずは Chrome / Edge を想定した実装で問題ない。
- DB 化 (SQLite 等) や、単一バイナリ配布、ネイティブアプリ化に必要な設計。
- テスト戦略の網羅的な設計。MVP では最低限の型 + 手動確認を主とし、テスト自動化は今回のスコープ外。

## Decisions

主要な決定を ADR 形式で列挙する。番号は将来 `ADR-XXX` として個別に参照できるよう保持する。

### ADR-001: 実行形態は「ローカルサーバ型 (ターミナル起動)」を採用する

- **文脈**: 個人ローカルの Web アプリで、データをローカルファイルとして保存したい。ブラウザ側でファイルを直接扱う手段は限られており、選択肢によって使い勝手・配布・ブラウザ縛りが大きく変わる。
- **選択肢**:
  - (A) ブラウザ完結型 + File System Access API (フォルダ選択で直接読み書き)
  - (B) ブラウザ完結型 + IndexedDB + 自動 export/import
  - (C) ローカルサーバ型 (`npm run dev` 相当をターミナルから起動)
  - (D) 単一バイナリ配布 (Tauri / Electron 等で「ダブルクリックで起動」体験)
- **決定**: (C) ローカルサーバ型を採用する。
- **理由**:
  - (A) は Chrome / Edge 限定 + 権限フローが煩雑。「どのブラウザでも使える」を残したい。
  - (B) は結局 export/import の見た目を隠しただけで、ファイル実体を素直に扱えない。
  - (D) は体験は良いが、MVP 段階でビルド/配布パイプラインを組む複雑さが割に合わない。
  - (C) は「起動する」ステップこそ残るが、Next.js の標準的な開発体験そのままで、ファイルシステムを素直に扱え、ブラウザ縛りもない。
- **却下した案の理由**:
  - (A): ブラウザ縛りと権限 UX を許容できない。
  - (B): ローカルファイルとしての実体を第一級で扱いたい要件と噛み合わない。
  - (D): MVP に対して過剰。将来必要になれば別提案で追加できる (コアは変わらない)。
- **トレードオフ**: 起動時にターミナルを 1 回開く必要がある。個人用途で許容する。

### ADR-002: フレームワークに Next.js (App Router) を採用する

- **文脈**: 単一プロセスで UI とデータ永続化の両方を担う必要がある。フロントとバックが分離した構成にすると、個人ローカル用途としては重い。
- **選択肢**:
  - (A) Next.js (App Router)
  - (B) Vite + React + 別途バックエンド (Hono / Fastify 等)
  - (C) Remix
  - (D) SvelteKit / Nuxt など他フレームワーク
  - (E) 素の React + Express 等の伝統構成
- **決定**: (A) Next.js (App Router) を採用する。
- **理由**:
  - 単一プロセスで UI + サーバ処理を扱える (Server Components / Server Actions)。
  - ファイル I/O を Server Component / Server Action から直接呼べる。
  - 個人ローカル用途でありながら、将来 UI が育っても耐えられる汎用性を持つ。
  - ユーザーが今回学習対象として選んだ (経緯: 「Next.js を使うなら」との明示的な指定)。
- **却下した案の理由**:
  - (B) / (E): フロントとバックの分離は個人ローカル MVP には過剰。
  - (C): 選択肢としてはあり得るが、ユーザー指定が Next.js のため。
  - (D): 同上 (エコシステム上の優劣というより指定による選定)。
- **トレードオフ**: MVP スコープに対して Next.js はやや大げさ。将来機能を足す前提のオーバースペックとして受容する。

### ADR-003: 言語は TypeScript、スタイルは Tailwind CSS を採用する

- **文脈**: 個人プロジェクトとはいえ Spec-First で育てていくため、リファクタ耐性と可読性を確保したい。
- **選択肢**:
  - 言語: (A) TypeScript / (B) JavaScript
  - スタイル: (X) Tailwind CSS / (Y) CSS Modules / (Z) styled-components 等 CSS-in-JS
- **決定**: TypeScript + Tailwind CSS。
- **理由**:
  - TypeScript は Next.js のデフォルトかつ標準。型で仕様が表現できるため、Spec-First と相性が良い。
  - Tailwind は Next.js の公式推奨で、ドキュメント量が多い。DnD やレイアウト調整で試行錯誤する MVP フェーズでは、クラス直書きが最も速い。
- **却下した案の理由**:
  - (B) JavaScript: 型安全性を捨てる合理性がない。
  - (Y) CSS Modules: 個別ファイルを増やす負荷が MVP には見合わない。
  - (Z) CSS-in-JS: RSC (React Server Components) との組み合わせで制約があり、MVP で採用するメリットが薄い。
- **トレードオフ**: Tailwind のクラス名が長くなり、JSX が読みづらくなる箇所が出る可能性がある。共通化は必要になったタイミングで行う。

### ADR-004: データ永続化は「JSON ファイル 1 枚」を採用する

- **文脈**: MVP のスコープは「単一板・3 列固定・タイトルのみ」で、扱うデータ量は少なく、複雑なクエリは不要。
- **選択肢**:
  - (A) JSON ファイル 1 枚 (`./data/tasks.json`)
  - (B) SQLite (`better-sqlite3` 等)
  - (C) LowDB / NeDB などの軽量埋め込み DB
  - (D) IndexedDB (ブラウザ側)
- **決定**: (A) JSON ファイル 1 枚を採用する。
- **理由**:
  - MVP のデータ構造が単純で、テーブルやクエリを必要としない。
  - JSON は目視で編集でき、破損時のリカバリが容易。
  - 実装量が最も少なく、Server Action からの読み書きが 1 関数で書ける。
  - Git 管理する場合の可視性も高い (今回は `.gitignore` 対象だが、必要になれば管理下に置き換えるのも容易)。
- **却下した案の理由**:
  - (B): 検索や大量データが必要になってから採用すればよい。今は過剰。
  - (C): SQLite の代替として使うほどの魅力がなく、Node.js 界での安定した第一選択とも言い難い。
  - (D): ADR-001 で採用しなかった「ブラウザ完結型」の系譜で、実行形態と整合しない。
- **トレードオフ**:
  - 同時書き込みや大量データに弱い (シングルユーザー・ローカル前提なので許容)。
  - 将来、検索・フィルタ・複数板を導入する際に SQLite への移行提案が必要になる可能性が高い。この移行余地は認識しておく。

### ADR-005: データ保存先は `./data/tasks.json` (リポジトリ配下) とする

- **文脈**: JSON ファイルの物理配置場所には複数の選択肢がある。
- **選択肢**:
  - (A) リポジトリ配下 `./data/tasks.json` (`.gitignore` 済)
  - (B) ユーザーホーム `~/.task-board/tasks.json`
  - (C) 環境変数 `TASK_BOARD_DATA_PATH` などで切替可能に
- **決定**: (A) `./data/tasks.json` (リポジトリ配下、`.gitignore` 対象) を採用する。ユーザーからの明示的な指定がある。
- **理由**:
  - 開発中とユーザー利用の切り分けが不要 (常に同じ場所)。
  - 実データの所在が把握しやすい。
  - ホームディレクトリのフォルダ作成といった副作用が発生しない。
- **却下した案の理由**:
  - (B): 「アプリのデータ」としては筋が良いが、個人ローカルでリポジトリを 1 つしか持たない前提なら利点が薄い。
  - (C): MVP に対してオーバーエンジニアリング。必要になれば別提案で追加。
- **トレードオフ**: リポジトリを移動・複製するとデータも巻き込まれる。誤って `git add` しないよう `.gitignore` の徹底が必要。

### ADR-006: 書き込み系 API は Server Actions に統一する

- **文脈**: Next.js App Router では書き込み系 API を Server Actions (`"use server"`) と Route Handler (`app/api/.../route.ts`) の 2 通りで実装できる。
- **選択肢**:
  - (A) Server Actions に統一
  - (B) Route Handler (REST) に統一
  - (C) 併用 (混在)
- **決定**: (A) Server Actions に統一する。
- **理由**:
  - App Router 時代の「Next.js らしい」書き方で、フォーム / 変更操作を Client Component から型付きで直接呼べる。
  - MVP 段階で外部から叩く API が要らないため、REST エンドポイントを設計するコストが不要。
  - 認証・CORS など Route Handler で考慮すべき論点をスキップできる。
- **却下した案の理由**:
  - (B): 個人ローカル・単一プロセス構成で REST を切る合理性がない。
  - (C): 一貫性を優先する。混在は後から必要になれば ADR で追記する。
- **トレードオフ**: 外部プロセスや別クライアント (CLI 等) からデータを操作したくなった場合、Server Actions は直接呼べないため Route Handler 化が必要になる。MVP では発生しないと判断。

### ADR-007: ドラッグ & ドロップは `@dnd-kit` を採用する

- **文脈**: MVP スコープにカラム間のカード移動を DnD で実装することが含まれている (「できたら入れたい」→ 実装量的に MVP に含める判断)。
- **選択肢**:
  - (A) `@dnd-kit` (core + sortable)
  - (B) `react-dnd`
  - (C) `react-beautiful-dnd` (メンテナンス終了)
  - (D) ネイティブ HTML5 DnD API を素で使う
- **決定**: (A) `@dnd-kit` を採用する。
- **理由**:
  - 現在の React エコシステムで最も活発にメンテナンスされている DnD ライブラリ。
  - カンバンのカラム間並び替えは公式サンプル / 有名記事が豊富で、教科書的なユースケース。
  - キーボード操作・アクセシビリティが標準で考慮されている。
- **却下した案の理由**:
  - (B): API が古めで、学習コストの割にメリットが少ない。
  - (C): メンテ終了。新規採用の理由がない。
  - (D): 実装量が跳ね上がる。個人 MVP でこの車輪を再発明する意味はない。
- **トレードオフ**: 依存が 1 つ増える。バンドルサイズは個人ローカル用途で問題にならない。

### ADR-008: カンバンのカラム構成は「3 列固定 (Todo / Doing / Done)」でハードコードする

- **文脈**: 将来的にカラムのカスタマイズを入れる可能性はあるが、MVP では設定 UI もマイグレーション設計も要らない。
- **選択肢**:
  - (A) 3 列固定・コード側で列名も定数化
  - (B) 設定ファイルで列を定義できるようにする
  - (C) 実行時にユーザーが列を追加・並び替えできるようにする
- **決定**: (A) 3 列固定でハードコードする。
- **理由**:
  - Non-goals 通り、カラムカスタマイズはスコープ外。
  - データモデルを最小化でき、`tasks.json` の構造が明快になる。
  - 将来 (B) / (C) に拡張する余地は残しつつ、今作らないことでスコープを守る。
- **却下した案の理由**:
  - (B) / (C): Non-goals に該当。将来提案で扱う。
- **トレードオフ**: 将来カスタマイズを入れる際、データ構造 (`{ todo: [], doing: [], done: [] }` から `{ columns: [{ id, name, cards: [] }, ...] }` 等へ) の移行が必要になる。今の構造から変更する事実は認識しておく。

### ADR-009: カード並び順は配列の順序で表現する

- **文脈**: `./data/tasks.json` 内で各カラムのカードの並び順をどう表現するか。カード追加・削除・DnD 移動で頻繁に並び順が変わる。
- **選択肢**:
  - (A) 配列の並び順そのものを表示順とする (各カード要素は `id` と `title` のみ持つ)
  - (B) 各カードに `order` フィールド (数値等) を持たせ、それでソートする
- **決定**: (A) 配列の並び順そのものを表示順とする。
- **理由**:
  - データ構造が単純で、JSON を目視で読んだ時にそのまま並び順が把握できる。
  - 並び替え時の実装が `Array.prototype.splice` 等の配列操作で完結し、副次的なフィールドの再計算が要らない。
  - `order` 値の重複・欠番といった状態不整合の余地が原理的に存在しない。
  - `Requirement: 永続化スキーマ` (task-persistence) の記述と整合する。
- **却下した案の理由**:
  - (B): RDB (SQLite 等) への移行時には `order` 型の親和性が高いが、それは移行時点で改めて設計するのが筋。MVP 段階で先取りする合理性は薄い。並び替え時に「order 値をどう振り直すか」を別途決める必要が生じ、実装量も判断も増える。
- **トレードオフ**: 将来 DB 化する際は、配列順を `ORDER BY` 用のカラムに写す変換ステップが必要になる。ただしそもそも spec 書き換えを伴う移行のため、追加コストとしては小さい。

### ADR-010: 読み込み失敗時は専用 error.tsx でファイル状態を提示する

- **文脈**: `./data/tasks.json` の JSON パース失敗またはスキーマ不一致時、システムは黙って上書きせずエラー停止する方針 (task-persistence spec)。ユーザー (= 自分自身) に「何が起きたか」を分かる形で伝える UI が必要。
- **選択肢**:
  - (A) 素の Next.js エラーページに任せる (`error.tsx` を用意せず throw を上まで通す)
  - (B) App Router の `error.tsx` を用意し、対象ファイルパスとエラー種別を提示する
  - (C) 復旧 UI (再読み込みボタン、ファイル内容の一部プレビュー、リカバリ操作等) まで作り込む
- **決定**: (B) `error.tsx` を用意し、最小限の判別情報を提示する。
- **理由**:
  - JSON 破損が発生した際、ユーザーが原因ファイルを即座に特定できる状態にしておく方が復旧が早い。
  - (A) では「どのファイルの問題か」を毎回思い出す必要があり、個人ツールでも認知コストが積み重なる。
  - (C) はエラー発生頻度に対して実装量が過剰。MVP のスコープを守る。
- **却下した案の理由**:
  - (A): 情報密度が低く、原因調査の手間が発生する。
  - (C): Non-goals のスコープ規律から逸脱。実際にファイル破損が頻発するなら別提案で追加する。
- **トレードオフ**: `error.tsx` を 1 ファイル増やす分の実装コストと、破損時のエラーメッセージ設計 (ファイルパスと問題種別の 2 情報) が必要になる。

### ADR-011: 起動スクリプトは `npm run dev` のみを提供する

- **文脈**: Next.js の慣例では `npm run dev` (開発サーバ, HMR あり) と `npm run build` + `npm start` (本番ビルド) の 2 系統がある。個人ローカル用途では両者の性能差は問題にならない。
- **選択肢**:
  - (A) `npm run dev` のみ (create-next-app のデフォルトのまま)
  - (B) `npm start` を別途用意する (本番ビルド前提の起動)
  - (C) 両方用意する (dev = 開発用、start = 通常起動用)
- **決定**: (A) `npm run dev` のみ。
- **理由**:
  - MVP スコープにおいて、本番ビルドを行うほどの性能要求は存在しない。
  - スクリプトを増やすと「どちらで起動すべきか」の説明が README に必要になり、個人ツールとして冗長。
  - 後から `npm start` を足すのは proposal を要さない軽い変更であり、必要が生じてから追加すれば足りる。
- **却下した案の理由**:
  - (B) / (C): 現時点で本番ビルドを常用する動機がない。将来必要が生じた時点で判断すればよい。
- **トレードオフ**: 起動時間や実行時性能が本番ビルドに劣るが、個人ローカル用途で許容範囲。

## Risks / Trade-offs

- **[Risk] JSON ファイルの同時書き込み**: 単一プロセスなので通常は発生しないが、Next.js の HMR (Hot Module Reload) や複数タブからの操作が重なると理論上競合の余地がある。
  → **Mitigation**: MVP では Server Action 内で読み込み → マージ → 書き込みを 1 関数にまとめ、書き込みは同期的に実施。競合の温床となるロック機構は導入しない (シングルユーザー前提)。

- **[Risk] `./data/tasks.json` の誤コミット**: `.gitignore` から漏れると個人データがリモートに上がるリスク。
  → **Mitigation**: 初期セットアップ時に `.gitignore` に `data/` を明示的に含める。tasks で明示ステップにする。

- **[Risk] JSON 破損時のふるまい**: 手編集や書き込み中断で JSON が不正になる可能性。
  → **Mitigation**: 読み込み時に JSON パースが失敗した場合、空の初期状態にフォールバックせず、明確なエラーで停止する (データを黙って消さない)。復旧はユーザー自身のファイル編集に委ねる。

- **[Risk] 将来の DB 移行コスト**: JSON → SQLite 等への切り替え時、`task-persistence` capability の spec を書き換える必要がある。
  → **Mitigation**: 永続化ロジックを 1 モジュールに閉じ込め、UI / Server Actions からは関数境界越しでのみアクセスする。移行時の変更範囲を最小化する。

- **[Trade-off] Chrome / Edge 前提のブラウザ検証**: MVP では他ブラウザでの動作確認はしない。個人利用のため許容。

- **[Trade-off] Next.js のオーバースペック性**: MVP のためだけなら Next.js は重い。将来機能拡張の投資として受け入れる。
