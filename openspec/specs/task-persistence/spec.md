# task-persistence

## Purpose

カンバンボードの状態 (各カラムのカードと並び順) をローカル JSON ファイルに永続化し、開発サーバの再起動を跨いでボード状態を保持する。JSON ファイルの破損時にはユーザーが検査・修復できる状態を保ち、黙って上書きしない。

## Requirements

### Requirement: JSON ファイルによるローカル永続化

システムは、ボードの状態 (各カラムに属するカードとその並び順) をローカルファイルシステム上の JSON ファイルに永続化 SHALL する。

#### Scenario: 変更操作の後にファイルが更新される

- **WHEN** ユーザーがカードの追加・削除・移動のいずれかを実行する
- **THEN** システムはボードの最新状態を JSON ファイルに書き込む
- **AND** 書き込みは操作の応答が返る前に完了している

---

### Requirement: 保存先の固定

システムは、永続化先を リポジトリ配下の `./data/tasks.json` として固定 SHALL する。パスは環境変数や設定ファイルで変更できないものとする。

#### Scenario: 保存先パスがハードコードで固定されている

- **WHEN** システムが JSON ファイルを読み書きする
- **THEN** 対象パスは常にリポジトリルート配下の `./data/tasks.json` である
- **AND** 環境変数などによるパス上書き機構は存在しない

---

### Requirement: 保存ディレクトリの自動生成

システムは、`./data/` ディレクトリまたは `./data/tasks.json` が存在しない状態で書き込みを行う場合、ディレクトリとファイルを自動的に生成 SHALL する。

#### Scenario: 初回書き込みでディレクトリとファイルが作成される

- **WHEN** `./data/` ディレクトリが存在しない状態で初めての書き込みが発生する
- **THEN** システムは `./data/` ディレクトリを作成する
- **AND** `./data/tasks.json` を作成し、有効な JSON として最新のボード状態を書き込む

#### Scenario: ファイルのみ存在しない場合も同様に生成される

- **WHEN** `./data/` は存在するが `./data/tasks.json` が存在しない状態で書き込みが発生する
- **THEN** システムは `./data/tasks.json` を新規作成し、有効な JSON として最新状態を書き込む

---

### Requirement: 未初期化状態の読み込み

システムは、`./data/tasks.json` が存在しない状態で読み込みが要求された場合、`Todo` / `Doing` / `Done` すべてが空である初期ボード状態を SHALL 返す。ただしファイル自体はこの時点で作成 MUST NOT (最初の書き込みまで作成しない)。

#### Scenario: ファイルが存在しない状態の初回読み込み

- **WHEN** `./data/tasks.json` が存在しない状態でボード状態が読み込まれる
- **THEN** システムは 3 カラムすべてがカード 0 枚の状態としてボードを返す
- **AND** この読み込み処理は `./data/tasks.json` を作成しない

---

### Requirement: JSON 破損時の安全な失敗

システムは、`./data/tasks.json` の内容が有効な JSON としてパースできない、または期待するスキーマに合致しない場合、黙って上書き MUST NOT。読み込み処理を明示的なエラーで停止 SHALL し、ユーザーがファイル状態を検査・修復できる状態を保つ。

#### Scenario: パース失敗時にエラー停止する

- **WHEN** `./data/tasks.json` の内容が JSON としてパースできない
- **THEN** システムはボード状態を読み込めず、エラーを発生させる
- **AND** システムはその状態でファイルを上書きしない

#### Scenario: スキーマ不一致時にエラー停止する

- **WHEN** `./data/tasks.json` は有効な JSON だが、期待するスキーマ (3 カラムそれぞれのカード配列を含む構造) と一致しない
- **THEN** システムはボード状態を読み込めず、エラーを発生させる
- **AND** システムはその状態でファイルを上書きしない

---

### Requirement: 読み込み失敗時の専用エラー画面

システムは、ボード読み込みが失敗した場合 (JSON パース失敗またはスキーマ不一致)、自動生成のデフォルトエラー画面ではなく、原因の判別が可能な専用エラー画面をユーザーに表示 SHALL する。エラー画面は、少なくとも対象ファイルのパスと、発生している問題の種別 (JSON パース不能 / スキーマ不一致) が判別できる情報を含む MUST。

#### Scenario: JSON パース失敗時に専用エラー画面を表示する

- **WHEN** ボード読み込み時に `./data/tasks.json` の JSON パースが失敗する
- **THEN** システムは対象ファイルパスと「JSON パース不能」であることを示す専用エラー画面を表示する
- **AND** カード操作 UI (カード表示、追加フォーム、削除ボタン、DnD 領域等) は表示しない

#### Scenario: スキーマ不一致時に専用エラー画面を表示する

- **WHEN** `./data/tasks.json` はパース可能だが期待するスキーマと一致しない
- **THEN** システムは対象ファイルパスと「スキーマ不一致」であることを示す専用エラー画面を表示する
- **AND** カード操作 UI は表示しない

---

### Requirement: 永続化スキーマ

永続化 JSON は、最上位オブジェクトに `todo`, `doing`, `done` の 3 つのキーを持ち、それぞれの値がカードオブジェクトの配列である SHALL。カードオブジェクトは少なくとも一意な `id` (文字列) と `title` (文字列) を持つ。配列の並び順がそのままカラム内のカード表示順を表す。

#### Scenario: 空のボードは 3 キーがそれぞれ空配列のオブジェクトで表現される

- **WHEN** ボードが完全に空の状態でファイルに書き込まれる
- **THEN** ファイル内容は `{"todo": [], "doing": [], "done": []}` と等価な構造である

#### Scenario: カード追加後の JSON 構造

- **WHEN** `Todo` に 1 枚のカードが存在する状態で書き込まれる
- **THEN** ファイル内の `todo` 配列は 1 要素のオブジェクトを含む
- **AND** そのオブジェクトは `id` と `title` のフィールドを持つ

---

### Requirement: 永続化ロジックの分離

システムは、JSON 読み書きの実装を単一のモジュールに閉じ込め、UI・Server Actions 以下のレイヤーからは関数境界越しにのみ SHALL アクセスする。将来 SQLite などへ移行する際の変更範囲を最小化するためである。

#### Scenario: UI 層から直接 fs API を呼ばない

- **WHEN** カード操作を担う UI コンポーネントまたは Server Action が永続化を必要とする
- **THEN** それらは永続化モジュールが公開する関数を経由して読み書きを行う
- **AND** UI / Server Action のコード内で Node.js の `fs` モジュールや `./data/tasks.json` を直接参照 MUST NOT
